# Org Content Hub

**Status:** ✅ Active
**Last Updated:** 2026-06-12
**Audience:** Maintainers & AI agents (technical tier)

## Overview

The content hub is a **federated** model, not a content warehouse. Every
repository in the organization publishes **its own** GitHub Pages project site,
rendered with the shared zer0-mistakes theme via `remote_theme`. Content never
leaves its source repo. This repository contributes two things:

1. **The theme** every org site renders with (`remote_theme: <org>/zer0-mistakes`).
2. **The dashboard** at `/hub/` that presents, monitors, and links to all the
   sites — driven by metadata gathered from the GitHub API, not by copying
   content.

```text
  year-of-ai/2005 ──remote_theme──┐
  year-of-ai/2006 ──remote_theme──┤      this repo (zer0-mistakes)
  year-of-ai/2007 ──remote_theme──┼────▶ • theme (layouts/includes/assets)
  year-of-ai/2008 ──remote_theme──┘      • /hub/ dashboard + nav (metadata only)

  each repo builds & serves itself:
    year-of-ai.github.io/2005/   year-of-ai.github.io/2006/   …
```

### Why federated, not replicated / submodules

- **Each repo keeps its own URL.** Project Pages give `<org>.github.io/<repo>/`
  out of the box — exactly the desired scheme. Replicating content into this
  repo would publish it under *this* site's URL instead.
- **No staleness.** Submodules pin a commit SHA; the hub would lag until a bot
  bumped pointers. With `remote_theme`, each site always builds from its own
  `HEAD`, and the dashboard reflects live API state.
- **Single source of truth.** Editing happens in the source repo; there is no
  generated copy to drift or to accidentally edit.

## Components

| Piece | Path | Role |
|---|---|---|
| Registry | `_data/hub.yml` | Source of truth: org, discovery rules, Pages settings, exclude globs, per-repo overrides |
| Shared lib | `scripts/lib/hub.rb` | Registry load/validate, `gh` discovery, content-path + URL helpers |
| Metadata sync | `scripts/sync-hub-metadata.rb` (+ `.sh`) | API-only: regenerates the two dashboard data files below — clones nothing |
| Provisioner | `scripts/provision-org-sites.rb` (+ `.sh`) | Rolls the Pages scaffold out to org repos (PR by default) and optionally enables Pages |
| Scaffold templates | `templates/org-site/*` | `_config.yml` + `_data/navigation/main.yml` rendered into each org repo |
| Repo index | `_data/hub_index.yml` | Generated per-repo stats (pages, sections, Pages status, links) |
| Sidebar nav | `_data/navigation/hub.yml` | Generated tree for the dashboard (`sidebar.nav: hub`) |
| Dashboard | `pages/hub.md` → `/hub/` | Cards per repo, live/pending status, links to each site |
| Automation | `.github/workflows/hub-sync.yml` | Daily metadata refresh; commits only when the org changed |

## How a repo becomes a site

Run the provisioner from this (theme) repo:

```bash
./scripts/provision-org-sites.sh --dry-run            # preview the scaffold + PRs
./scripts/provision-org-sites.sh                      # open a scaffold PR per repo
./scripts/provision-org-sites.sh --enable-pages       # also enable Pages (deploy from branch)
./scripts/provision-org-sites.sh --repos 2005 --direct # commit straight to one repo's default branch
```

For each repo it clones the repo, inspects its markdown layout, renders
`templates/org-site/*` (filling in org/repo/title/description and a navigation
tree built from the repo's sections), and opens a PR adding:

- **`_config.yml`** — `remote_theme: <org>/zer0-mistakes`, `baseurl: /<repo>`,
  `url: https://<org>.github.io`, and the GitHub Pages default plugins
  (`optional-front-matter`, `readme-index`, `default-layout`,
  `titles-from-headings`, `relative-links`) so plain markdown renders with no
  front-matter requirements.
- **`_data/navigation/main.yml`** — navbar/sidebar entries for the repo's
  top-level pages and section folders.

Safety rails: a repo whose `_config.yml` was **not** generated by this script
is skipped (never clobbers a hand-rolled site config), and re-running is
idempotent (identical scaffold ⇒ no new PR/commit). The generated `_config.yml`
carries a `zer0-mistakes org hub scaffold` marker used to detect ownership.

## How the dashboard stays current

```bash
./scripts/sync-hub-metadata.sh               # refresh dashboard data from the API
./scripts/sync-hub-metadata.sh --dry-run     # preview
./scripts/sync-hub-metadata.sh --check       # validate registry + generated data (CI/PR gate)
```

The metadata sync calls the GitHub API for each registered repo — the file tree
(for page/section counts), the Pages endpoint (live vs. pending), and section
index titles — and regenerates `_data/hub_index.yml` and
`_data/navigation/hub.yml`. **No repo is cloned and no content is copied here.**
Output is deterministic, so the scheduled `hub-sync.yml` workflow only commits
(`chore(hub): refresh dashboard metadata`) when something actually changed,
which never triggers a version bump.

## Operations

- **Add a source repo:** nothing — auto-discovery picks up new non-archived org
  repos on the next metadata sync. Add an entry under `repos:` in `_data/hub.yml`
  only to override its title/branch, then run the provisioner to give it a site.
- **Remove a repo:** archive it (or add it to `exclude_repos`); the next sync
  drops it from the dashboard.
- **Change the theme for all sites:** push to this repo — every org site rebuilds
  against the new `remote_theme` on its next build.
- **First-time org rollout:** `./scripts/provision-org-sites.sh --enable-pages`,
  merge the PRs, then `./scripts/sync-hub-metadata.sh` to light up the dashboard.

## Design constraints

- Org repos must allow GitHub Pages and (for `remote_theme`) the theme repo must
  be public or accessible to them.
- The provisioner needs `gh` authenticated with repo (and, for `--enable-pages`,
  Pages) write scope on the org. The metadata sync needs only read scope.
- Only markdown content is counted/linked. Sites can carry their own assets;
  the dashboard does not inventory them.
